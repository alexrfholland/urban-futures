# Blender Template Spec

## 1. Purpose

This file defines the target heavy-scene Blender template for the 2026 workflow.

Use it to answer:

- what the template blend must contain
- what must be generated into it by scripts
- what must be rerun when state changes
- what should stay out of the template

This is a template spec, not a description of every current working blend. The current reference for the city pipeline is still:

- [`data/blender/2026/2026 futures heroes6.blend`](/Users/alexholland/Coding/volumetric-scenarios-rhino-bim-gia/data/blender/2026/2026%20futures%20heroes6.blend)

- [`data/blender/2026/2026 futures heroes6_baseline.blend`](/Users/alexholland/Coding/volumetric-scenarios-rhino-bim-gia/data/blender/2026/2026%20futures%20heroes6_baseline.blend)

## 2. Template Contract

The heavy-scene template must provide:

1. one scene per site, with the scene name matching the site name
2. the site cameras and camera archive collections
3. the live clip objects and camera proxy clipbox system
4. imported base-world point clouds
5. world-cube objects generated from the base-world point clouds
6. empty or generated scenario collections for the instancer outputs
7. materials, AOVs, and EXR output branches for the render contract
8. embedded copies of the operational scripts that are run from inside Blender

## 3. Required Scene Structure

### 3.1. View Layers

For the city template, the scene must contain:

- `pathway_state`
- `existing_condition`
- `city_priority`
- `city_bioenvelope`
- `trending_state`

### 3.2. Pass Indices

The template must preserve this pass-index contract:

| Pass index | Meaning |
| --- | --- |
| `1` | world / road / ground and cube derivatives |
| `2` | buildings / base built form |
| `3` | arboreal point clouds and instanced habitat features |
| `5` | bioenvelope geometry |

### 3.3. Top-Level Collections

For the city template, the scene must provide these top-level collections:

- `City_World`
- `City_WorldCubes`
- `City_Manager`
- `City_Camera`
- `City_Camera_Archive`
- `Year_city_180_positive`
- `Year_city_180_positive_priority`
- `Year_city_180_trending`
- `city_envelope`
- `Envelope_trending`

These three state collections are generated by the instancer:

- `Year_city_180_positive`
- `Year_city_180_positive_priority`
- `Year_city_180_trending`

The template should preserve the collection names even if the collections start empty.

### 3.4. Expected Objects

The city template should contain or support:

- base-world point clouds such as `city_buildings(.001)` and `city_highResRoad(.001)`
- cube-derived world objects such as `_cubes` variants
- live clip objects such as `City_ClipBox`
- site cameras in `City_Camera`
- archived cameras in `City_Camera_Archive`

## 5. State Change Workflow

When changing state in the template:

1. refresh the base-world PLYs if the site geometry changed
2. rerun [`final/_blender/2026/b2026_world_cubes.py`](/Users/alexholland/Coding/volumetric-scenarios-rhino-bim-gia/final/_blender/2026/b2026_world_cubes.py) if the world point clouds changed
3. rerun [`final/_blender/2026/b2026_transfer_vtk_sim_layers.py`](/Users/alexholland/Coding/volumetric-scenarios-rhino-bim-gia/final/_blender/2026/b2026_transfer_vtk_sim_layers.py) if state-linked world attributes changed
4. run [`final/_blender/2026/b2026_instancer.py`](/Users/alexholland/Coding/volumetric-scenarios-rhino-bim-gia/final/_blender/2026/b2026_instancer.py)
5. run [`final/_blender/2026/b2026_clipbox_setup.py`](/Users/alexholland/Coding/volumetric-scenarios-rhino-bim-gia/final/_blender/2026/b2026_clipbox_setup.py)
6. run [`final/_blender/2026/b2026_camera_clipboxes.py`](/Users/alexholland/Coding/volumetric-scenarios-rhino-bim-gia/final/_blender/2026/b2026_camera_clipboxes.py)
7. rerun [`final/_blender/2026/b2026_setup_view_layer_exr_outputs.py`](/Users/alexholland/Coding/volumetric-scenarios-rhino-bim-gia/final/_blender/2026/b2026_setup_view_layer_exr_outputs.py) if the compositor EXR branches are stale
8. render multilayer EXRs
9. hand those EXRs to the lightweight compositor workflow

Do not rely on the instancer auto-run flags for the clip scripts.

## 6. Template Should Exclude

The template should not carry:

- stale generated state geometry that is no longer part of the current contract
- temporary shadow helpers
- edge-lab experiment nodes
- one-off debug compositors
- baseline-specific overrides

## 7. Validation Checklist

Before calling a blend the template, confirm:

- the site scene exists and is named correctly
- the full view-layer contract exists
- the pass-index contract is intact
- the top-level collection contract is intact
- the world point clouds and world cubes are present
- the clip objects and camera clipbox system work
- the instancer builds `positive`, `positive_priority`, and `trending` into the correct collections
- the resource and envelope AOV setup is present
- the EXR output branches exist and write one multilayer EXR per view layer
- the embedded clip scripts are synced with disk
